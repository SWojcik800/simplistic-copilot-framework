# Tasks Directory

This directory holds active tasks. Each task is a **directory** containing a task definition, orchestrator state, and worktrees.

## Task Directory Structure

```
tasks/
  <task-id>/
    task.md                    # Task definition
    orchestrator_state.yaml    # Orchestration state (managed by orchestrator)
    worktrees/                 # Git worktrees for this task
      <repo>-<branch>/        # Checked-out worktree
```

## Task Definition Template

Create a directory `tasks/<task-id>/` and add a `task.md`:

```markdown
# Task: <title>

- **Repo:** <repo-name>
- **Type:** bugfix | feature | refactor | review | git-flow | initialize
- **Branch:** <branch-name>

## Description
<Describe what needs to be done.>

## Acceptance Criteria
- [ ] <criterion 1>
- [ ] <criterion 2>

## Notes
<Any additional context, links, or references.>
```

## Orchestrator State

The orchestrator automatically creates and manages `orchestrator_state.yaml` in each task directory. This file tracks:
- Current status (`pending`, `setting-up`, `in-progress`, `pr-pending`, `completed`, `blocked`, `failed`)
- Worktree path
- Step-by-step progress
- PR information (if applicable)
- Timestamps

Do not edit this file manually — it is managed by the orchestrator agent.

## Conventions

- One task per directory.
- Use descriptive task IDs: `my-app-add-user-auth`, `api-fix-login-crash`
- The orchestrator reads this directory to discover and manage work.
- Worktrees are created inside each task's `worktrees/` subdirectory.
- Remove or archive task directories after they are completed and merged.
